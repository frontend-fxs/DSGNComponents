FXStreet frontend architecture

The architecture is builded with the "Atomic design" approach.

The project is using Less.js. In order to ensure maintenance and scalability, each group of components are divided into different LESS files to import all into the final "fxs_screen.min.css" which is the main CSS for the public site.

The widgets for third parties compiles specific less files to generate one common minified CSS (core.min.css) and each widget is compiled in its own minified css (widgetname.min.css).

Less internal rules

Classes,# & tag styling

  • Design and front-end team only style by classes, never by ID (only IT can ad IDs).
  • Any created class must start with the prefix: fxs_ in order to avoid conflicts with third parties styles.
  • Use underscores (_) to split words in class names. This will help to find and distinguish easily FXStreet from vendor classes.
  • If an ID it’s needed for js reasons, ask IT team and comment the use of this ID
  • Avoid styles any tag directly, this ensure the maintenance of the code.
  • There’s a small set of HTML tags styled directly in the reset.less file. Those are the defaults for the website.

Vars

The main variables of the site must be on the vars.less file, this will ensure an easy maintainance.

The main variables:

  • Breakpoints
  • Grid system vars
  • Color vars
  • Typo vars
  • Measures, scales, spaces and box sizes
  • Bootstrap vars overrides

HEADS UP! All vars must be inside this file.

IMPORTANT! Try to insert all measures, colors, sizes, or any possible constant so, if a global change it's needed, it will be as simple as change the var parameters from here.

Extends

In the utilities.less file are contained all the generic classes:

  • Margins to push or pull objects
  • Modules fix measures
  • Transitions and animations
  • Formatted text styles
  • Generic modules, boxes or fix/hack classes
  • Display and mobile helpers (none, visible, hidden in xs screens, ...)

This classes will have CSS output so, keep it clean.

All of this classes can be used directly in the HTML, combine it with other classes to dictate properties in an object or can be extended in more complex objects so, once compiled, the common rules will be grouped in the final CSS.

HEADS UP! Only classes from this file can be extended. It’s a must: Don’t extend from any other files in order to avoid duplicity or long breadcrumbs to find the original CSS rule.

Widgets

All the components for third parties that must works in or outside the FXStreet website: price charts, rates, technical indicators, all the FXS studies charts.

For the third parties, there will be two CSS files to deliver: core.min.css and (each) widget.min.css

  • core.min.css

    Imports specific less files from the FXS website and force some styles nested in specific fxs_ classes to ensure correct view:

    The idea with this structure it’s to guarantee that the core css will be up to date with the fxs website. The core.min.css will be compiled every time one of the imported files is updated.

    HEADS UP! Don't style any HTML tag without a nesting rule within an .fxs_ class. This widgets are sold to third parties so, a direct style to an HTML tag may crash their website.

  • widget.min.css

    Specific styles for each widget.

    The widgets have different sizes in order to fit in as many layout cols as possible so, almost every widget has three specific classes with different breakpoints. They are organized as follows:

    • Common default styles: color, font styles, basic layout and chart styling, the common naming for each widget main class is .fxs_widget_widgetname this class combines with contextual classes to set the widget size:

      • .fxs_widget_mini: The smallest size, to fit in small sizes
      • .fxs_widget_default: The medium size, to fit in spaces = or > half page
      • .fxs_widget_big: The biggest size, to fit in fullpage spaces

Almost all chart widgets uses the c3 charts library. The CSS for this library is imported at the very beginning of the screen.less & core.less files. The overrides of c3 library are contained in the components.less file.

IMPORTANT! Each widget compiles into its own widget.min.css for third parties so, don’t extend classes in the widgets less files, it will works for FXStreet site (due to the @import in screen.less) but it won’t do anything on the standalone version of the widget

Try every widget on a standalone version without screen.css.min link.

Use of vendor components

Every third party CSS component or styles for js widgets will be in vendor’s folder.

Try to import only the part or the component that will be used: avoid waste of CSS. If possible, import only .less files so it can be compiles within one simple step -This will be explained later-.

HEADS UP! Don't edit directly from vendor files, override styles & vars into the FXS LESS files. By editing directly changes may be lost in case of updates.

Current vendor imports:

  • Bootstrap v3.3.6
    • component-animations.less
    • modals.less
    • tooltip.less

    HEADS UP! Overrides are contained in its own custom fxs less files: vars, mixins, components…

    HEADS UP2! Bootstrap is changing to a new version (4.0) with Sass so download the Github project of the v3.3.6 to have reference of all the components. In a near future, an update to Sass will be needed.

  • lesshat.less: Less mixins library. Read the docs, use it and save tons of writing ;)

Mobile first!

The FXStreet site is built with a mobile first approach, the CSS and HTML try to built a progressive enhancement experience this means that the first thing to build it’s the core content (HTML) and the basic styles to make sure of the correct visualization across all devices and user agents, then add the cool features and media queries for bigger screens.

Semantic usage of HTML5

The site is built in HTML5 and the correct use of tags must be a common practice: avoid “divitis”.

Use section, article, aside, header, footer, asides, address, ul, ol… be as semantic as possible with each type of content for markup.

HTML Tags reference: http://www.w3schools.com/tags/default.asp.

Object oriented CSS

Always write CSS with the Object Oriented CSS (OOCSS) approach. Means separate structure from skin => Layout ≠ visual features.

When writing think of small pieces that build big components when work together. Build reusable simple objects that can create more complex objects when combined.

IE: for styling a blue button, think of the future and use of small components, it could be more button styles with other colors, shapes or sizes so, separate the properties

  • Button common layout: .fxs_btn
  • Button color: fxs_btn_default
  • Button size: fxs_btn_big

Naming

Use semantic, descriptive, neutral and visual feature free names:

Don’t set a name by the color, fix measures or any visual feature.

Nesting rules

As shown before, the work of the front-end is to build small parts, components, “bricks” of the building. And each brick must be always a brick no matter where is it.

Avoid complex nesting rules, every component must work in every part of the site.

Like Lego

At last, a final tough: think of building CSS components like lego blocks.

Take blocks of different sizes and shapes to build the castle or a spaceship, but at the end, each block work with no dependency of other clocks or pieces. And, of course, can be reused in other constructions.

Front-end development mantras

  1. Always edit the less files: Never -under death penalty- edit directly from the CSS files (nor the extended or minified).
  2. The less files must remain without custom, ad hoc or temporally additions.
  3. For topic and landing pages -if it’s needed-, create their custom CSS file and add it via the CSS widget in Sitefinity editor, this will add the custom CSS in the head of the created page.
  4. Avoid complex nesting rules, each component most works correctly within all the site with no dependency of specific nesting rules.
  5. Avoid styling any HTML tag: all styles must depend from classes.
  6. Be careful with “divitis”: there are plenty of tags with specific use and with default behaviours.
  7. Comment everything: things may appear obvious until time and people pass.
  8. Communicate constantly with the designer and the front-end lead in order to make sure what to do before styling anything.
  9. Sketch, plan and make conceptual proofs with the team before writing any complex layout use tools as the browser’s inspector or apps like codepen. This will save valuable time.
  10. Respect the design but remain flexible: don’t be afraid to take design decisions if any design forces to complicate or twist too much the natural way of the HTML.
  11. Keep It Simple: The most important rule. Don’t override all rules each time, write the necessary lines, not more, not less, double or triple check your code and review it with all available front-ends in FXStreet.
